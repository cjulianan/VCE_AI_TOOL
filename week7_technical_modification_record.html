VCE AI Tool / Week 7

Technical Modification Record

Recommended changes for reliable multi-question handling, controlled data analysis, semantic retrieval, and future document integration.

Status: ProposedAudience: Project TeamDate: July 2026

1. Executive Summary

Recommendation: Do not begin with a fully autonomous data agent. First stabilize the data and metadata contracts, then change the application from a single-route pipeline into a structured multi-task pipeline. After that, add a small set of validated DuckDB tools for calculations. Embeddings and PDF retrieval should follow once routing and citation behavior are reliable.

The current prototype already provides useful foundations: a Shiny interface, metadata-based routing, locality lookup, DuckDB access, and an LLM connection through ellmer. The next phase should preserve that architecture while making its internal state explicit, testable, and capable of handling more than one dataset, locality, or operation per prompt.

2. Current Constraints

Routing

Single-result state

The current cache primarily retains one metadata path and one locality. This prevents reliable execution of prompts containing multiple questions, datasets, counties, or entity lookups.

Data contracts

Metadata inconsistencies

Some newly added datasets are not registered for chatbot routing, and several CSV, metadata, locality, and school-district mappings require alignment before automated analysis is added.

Calculation

LLM-dependent statistics

The prototype may occasionally infer percentages from retrieved records, but the language model is not a reliable calculation engine. Numerical operations should be executed by DuckDB.

Retrieval

Keyword-only matching

Keyword and string-similarity routing is useful for the prototype but will not scale well to varied wording, policy documents, and cross-domain questions.

3. Proposed Modifications

3.1 Stabilize the data and metadata layer

This work should be completed before adding agent behavior.

  • Adopt one required metadata schema for every routable dataset.
  • Register all approved datasets in the master registry, including the newly added health, mental-health, special-education, and state-testing data.
  • Validate metadata-to-CSV field names in both directions.
  • Replace user-specific absolute paths with project-relative paths.
  • Resolve county, independent-city, and multi-jurisdiction school-division mappings without silently dropping unmatched records.
  • Add a reproducible dependency manifest and automated contract checks.

3.2 Replace single-route state with a structured task plan

Each user prompt should first be converted into an array of independent tasks. A task should contain:

  • a stable task identifier and the original sub-question;
  • one or more candidate datasets;
  • zero or more localities, FIPS codes, or named entities;
  • an operation such as describe, lookup, compare, count, or average;
  • requested measures, filters, years, and grouping fields;
  • retrieved records, calculation results, and source information.
Prompt
  -> Structured task decomposition
  -> Independent dataset and locality routing
  -> Validated data execution per task
  -> Results labeled by task and source
  -> Final response synthesis in the user's original order

3.3 Add controlled data tools

A limited data-query agent is feasible within the existing stack. The model should select from a small set of application-owned functions rather than execute unrestricted SQL.

ToolPurposeRequired safeguards
search_dataset()Retrieve selected fields using approved filters and localities.Allowlisted datasets, fields, and operators.
find_entity()Locate a named hospital, school, facility, or other entity.Normalized matching, result limits, and explicit source.
summarize_data()Calculate count, sum, average, minimum, maximum, or grouped statistics.Validated numeric fields and deterministic DuckDB execution.
compare_localities()Run the same measure for multiple counties or cities.Consistent year, unit, denominator, and missing-data handling.

All statistics should be computed by DuckDB and returned with the dataset, locality, year, measure, unit, and calculation method. Percentages should identify the numerator and denominator whenever possible.

3.4 Add semantic retrieval in stages

  1. Embed dataset descriptions, column descriptions, keywords, geographic coverage, and temporal coverage.
  2. Use semantic similarity to produce candidate datasets, then apply deterministic eligibility checks.
  3. For PDFs, extract text, split it into page-aware chunks, attach source metadata, and retrieve only relevant chunks.
  4. Require page-level citations for document answers and dataset-level citations for tabular answers.

Embeddings should support retrieval and routing. They should not replace database calculations or exact filtering.

3.5 Strengthen safety and reproducibility

  • Escape or sanitize user input, model output, and imported chat history before HTML rendering.
  • Validate uploaded session files against an expected schema.
  • Use parameterized queries and metadata allowlists.
  • Add tests for routing, locality resolution, multi-question decomposition, calculation accuracy, and citations.
  • Create a fixed evaluation set of representative questions and expected results.

4. Concrete Multi-Question Examples

Example A: Two independent intents

User prompt: “What data do you have on crops? Also, what city is Bath Community Hospital in?”

[
  {
    "task_id": 1,
    "dataset_ids": ["nass_crops"],
    "operation": "describe"
  },
  {
    "task_id": 2,
    "dataset_ids": ["vgin_hospitals"],
    "entity": "Bath Community Hospital",
    "operation": "lookup"
  }
]

The second task requires entity-name lookup, not county-first filtering. Adding only a locality loop would not solve this case.

Example B: One intent across two localities

User prompt: “What disease data do you have for Roanoke County? What about Fairfax County?”

{
  "task_id": 1,
  "dataset_ids": ["vdh_diseases"],
  "localities": [
    {"name": "Roanoke County", "fips": "51161"},
    {"name": "Fairfax County", "fips": "51059"}
  ],
  "operation": "compare"
}

The same query definition should be applied to both localities, and the final answer should clearly separate or compare the results using consistent years and units.

5. Recommended Delivery Sequence

PhaseScopeCompletion evidence
1. StabilizeMetadata schema, registry, paths, locality mappings, validation.All approved datasets pass automated contract checks.
2. Multi-task MVPQuestion decomposition, task-list cache, entity and multi-locality routing.Both Week 7 examples return correctly separated answers.
3. Data toolsValidated lookup, count, sum, average, and comparison operations.Results match independently calculated reference values.
4. Semantic retrievalMetadata embeddings and hybrid candidate ranking.Improved routing on paraphrased evaluation questions.
5. DocumentsPDF ingestion, chunking, retrieval, and page citations.Answers cite the correct document and page.

6. Decisions Requested from the Team

  1. Which statistical operations are required for the final demonstration?
  2. Should the MVP answer independent sub-questions only, or also perform cross-dataset comparisons?
  3. How should multi-jurisdiction school divisions be represented in county-level answers?
  4. Which datasets are approved and complete enough to enter the routing registry?
  5. What evaluation questions and accuracy criteria will define a successful prototype?

7. Conclusion

The recommended path is incremental: make retrieval reliable, make the pipeline multi-task, add deterministic calculations, and then expand to semantic and multimodal retrieval. This preserves the team’s existing work, reduces implementation risk, and produces a testable minimum viable system before advanced agent behavior is attempted.
